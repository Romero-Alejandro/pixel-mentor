# Design: Step Type Unification

## Technical Approach

The current implementation suffers from a fractured definition of `StepType`. The database uses an `UPPERCASE` enum (`CONTENT`, `ACTIVITY`), the domain layer uses a `lowercase` string literal type (`'content'`, `'activity'`), and the application layer attempts to support both, leading to inconsistency and bugs.

This design unifies the `StepType` definition around a single source of truth: the Prisma-generated enum. The refactor will involve:

1.  Establishing the Prisma `StepType` enum as the canonical definition for the entire backend.
2.  Eliminating the manually defined `StepType` in the domain layer and replacing it with one derived from the Prisma enum.
3.  Refactoring all use cases, services, and tests to use only the canonical `StepType` enum for comparisons and assignments, removing all checks against lowercase string literals.
4.  Introducing a mapping or adapter at the API boundary if the frontend relies on lowercase types, though the primary goal is to enforce consistency internally first.

## Architecture Decisions

### Decision: Adopt Prisma Enum as the Single Source of Truth

- **Choice**: The `StepType` enum generated by Prisma from `schema.prisma` will be the one and only source of truth for step types within the backend.
- **Alternatives considered**:
  1.  Using lowercase strings everywhere and mapping at the database layer. This would fight against Prisma's conventions and require manual casting on every database operation, which is error-prone.
  2.  Maintaining both and continuing to check for both. This is the status quo and is the source of the problem.
- **Rationale**: The database schema is the ultimate contract. By using the Prisma-generated enum, we get type safety, auto-completion, and a single point of definition that propagates from the database up through the application layers. This eliminates the possibility of typos and inconsistencies between layers.

### Decision: Remove Redundant Domain Type Definition

- **Choice**: The manual `export type StepType = 'content' | ...` in `apps/api/src/features/recipe/domain/entities/recipe.entity.ts` will be deleted.
- **Alternatives considered**: Keeping the domain type and trying to keep it in sync with the Prisma enum manually. This is brittle and unnecessary.
- **Rationale**: The domain entity should reflect the data structure, and the `StepType` is defined by Prisma. We will import the `StepType` enum directly from the generated Prisma client where needed. This simplifies the domain and removes a source of potential drift. If a value object is desired, it should wrap the Prisma enum.

## Data Flow

The data flow remains largely the same, but the `stepType` property will now be consistently `UPPERCASE` at all stages within the backend.

```text
1. API Request (DTO)
       │
       ▼
2. Application Layer (Use Case)
   - Reads RecipeStep from Repository (Prisma)
   - stepType is now ALWAYS `StepType.CONTENT`, etc.
   - All logic (`requiresStudentInput`, `extractStaticContent`) is simplified
     to only check against the enum.
       │
       ▼
3. Domain Layer (Entity)
   - The `stepType` property is typed with the Prisma enum.
       │
       ▼
4. Infrastructure Layer (Prisma Repository)
   - Persists `StepType` enum value directly to the database.
```

## Interfaces / Contracts

The primary interface change is within the domain entity.

**Before:**

```typescript
// in recipe.entity.ts
export type StepType = 'content' | 'activity' | 'question' | 'intro' | 'closure';

export interface RecipeStep {
  // ...
  readonly stepType?: StepType;
}
```

**After:**

```typescript
// in recipe.entity.ts
import type { StepType as PrismaStepType } from '@/database/generated/client';

export type StepType = PrismaStepType; // Or just import and use PrismaStepType directly

export interface RecipeStep {
  // ...
  readonly stepType?: StepType;
}
```

_(Note: The exact implementation can be to alias the type or use it directly, but the manual definition will be gone.)_

The Prisma `StepType` enum will now be the standard.

```typescript
// From Prisma
export declare const StepType: {
  CONTENT: 'CONTENT';
  ACTIVITY: 'ACTIVITY';
  QUESTION: 'QUESTION';
  INTRO: 'INTRO';
  CLOSURE: 'CLOSURE';
  EXAM: 'EXAM';
};
export type StepType = (typeof StepType)[keyof typeof StepType];
```

## Testing Strategy

| Layer       | What to Test                          | Approach                                                                                                                                                                                                                                |
| ----------- | ------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Unit        | `orchestrate-recipe.use-case`         | Verify that the refactored `requiresStudentInput` and `extractStaticContent` functions behave correctly using only the `StepType` enum. Ensure tests that previously used lowercase strings now fail and are updated.                   |
| Unit        | Other Use Cases & Services            | Update mocks and assertions in all related tests to use the canonical enum.                                                                                                                                                             |
| Integration | API Endpoints (`/recipes/{id}/steps`) | If API contracts are changed (i.e., from lowercase to uppercase), integration tests must be updated to reflect the new expected output. If a mapping layer is added to preserve the old contract, tests must verify it works correctly. |
| E2E         | N/A                                   | No E2E tests are expected to be needed for this internal refactoring, unless it exposes a change to the frontend client.                                                                                                                |

## Migration / Rollout

No data migration is required. The database already stores the values in the correct `UPPERCASE` format. This is purely a code-level refactoring. The change can be rolled out in a single deployment.

## Open Questions

- [ ] Does the frontend expect lowercase `stepType` values? If so, we must add a mapping layer in the API response DTOs to convert the `UPPERCASE` enum values back to `lowercase` strings to avoid breaking the client. This should be investigated before implementation.
